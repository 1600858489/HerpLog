# 爬宠饲养记录工具设计方案（鳞记）

## 一、项目目标

本项目定位为**个人长期使用的爬宠饲养记录工具**。

设计目标不是构建一个复杂的管理系统，而是打造一个能够**每天坚持使用数年**的记录工具。

核心设计原则：

> **记录尽可能简单，提醒尽可能智能。**

整个系统围绕三个问题展开：

1. **今天应该做什么？**（Today）
2. **今天发生了什么？**（Timeline）
3. **下一步应该什么时候做？**（Rule Engine）

---

# 二、设计原则

## 1. Record First

记录是整个产品最重要的功能。

任何记录都应遵循：

> **3~5 秒完成。**

尽可能减少：

* 输入框
* 页面跳转
* 重复填写
* 用户思考

软件应该帮助用户记忆，而不是让用户填写表格。

---

## 2. Timeline First

整个软件围绕**时间线（Timeline）**设计。

而不是：

```
喂食
体重
蜕皮
排泄
```

而是：

```
小黑

6月26日
🍖 喂食

6月28日
💩 排泄

7月5日
⚖️ 称重

7月12日
🧤 蜕皮
```

所有事件天然按时间排序。

时间线即宠物成长记录。

---

## 3. Today First

首页只关心：

> 今天应该做什么。

首页不展示复杂统计。

例如：

```
今天

🐍 小黑
建议喂食（已超过1天）

🦎 阿黄
今天喷雾

🐢 小绿
建议称重
```

完成后自动消失。

---

## 4. Zero Thinking

尽可能做到：

> 用户只记录"变化"，不用重复记录"状态"。

例如：

无需填写：

* 日期（默认今天）
* 食物（默认上次）
* 重量（默认推荐）
* 时间（自动记录）

真正需要输入的内容应该只有：

> 今天与昨天相比发生了什么。

---

# 三、整体架构

整个系统由三个核心模型组成。

```
Pet
│
├── Event（事件）
│
├── State（状态）
│
└── Care Plan（饲养方案）
```

职责完全独立。

---

# 四、Pet（宠物）

每只宠物拥有独立档案。

例如：

```
小黑

物种

品系

性别

出生日期

购入日期

照片
```

这些信息几乎不会频繁变化。

因此不会重复填写。

---

# 五、Event（事件）

所有历史记录均属于 Event。

例如：

```
Feed

Weight

Poop

Shed

Medicine

Photo

Environment

Custom
```

以后新增任何记录：

```
换垫材

换水

冬化开始

繁殖

运输

打针
```

都只是新增 Event Type。

数据库无需修改结构。

---

## Event 时间线

例如：

```
今天

🍖 吃了18g冻鼠

↓

两天后

💩 排泄

↓

四天后

⚖️ 460g

↓

十天后

🧤 蜕皮
```

所有事件自然形成成长时间线。

---

# 六、State（状态）

事件描述：

> 发生了什么。

状态描述：

> 当前持续处于什么状态。

例如：

```
正常

冬化

拒食

治疗中

繁殖中

隔离观察
```

状态具有：

```
开始时间

结束时间
```

例如：

```
冬化

2026-11-01

↓

2027-02-20
```

状态会直接影响提醒规则。

例如：

冬化：

```
暂停喂食提醒
```

治疗中：

```
增加用药提醒
```

---

# 七、Care Plan（饲养方案）

提醒不采用固定日期。

而采用：

> Rule Engine

每只宠物拥有一个饲养方案。

例如：

```
幼体玉米蛇

推荐：

5~7天

食物：

冻鼠

推荐重量：

体重10~15%
```

或者：

```
成年豹纹守宫

每3天

8~10只昆虫
```

所有提醒全部实时计算。

---

# 八、提醒系统（Rule Engine）

系统不会保存：

```
next_feed_date
```

而是实时计算。

例如：

```
last_feed

+

Care Plan

↓

计算

↓

今天是否提醒
```

这样：

修改方案后无需重新计算全部数据。

---

## 提醒等级

建议采用三级提醒。

```
绿色

未到时间
```

```
黄色

进入推荐区间
```

```
红色

超过推荐周期
```

例如：

```
推荐：

5~7天

第4天

绿色

↓

第5天

黄色

↓

第8天

红色
```

---

# 九、记录设计

采用：

> Action Flow

而不是传统表单。

例如：

点击：

```
🍖 喂食
```

第一步：

```
今天喂了吗？

✅ 吃了

❌ 拒食
```

第二步：

```
还是：

18g 冻鼠？

✅

修改
```

点击：

```
保存
```

结束。

整个过程：

```
🍖

↓

✅

↓

✅

↓

完成
```

通常只需几秒。

# 二十、总结

本方案的核心思想可以概括为四句话：

1. **时间线优先（Timeline First）**：所有记录按时间组织，每只宠物都有完整的成长历史。
2. **今日优先（Today First）**：首页只展示今天需要完成的事项，而不是堆砌统计信息。
3. **规则驱动（Rule Engine）**：提醒基于饲养方案和当前状态实时计算，而不是依赖固定的提醒日期。
4. **低成本记录（Zero Thinking）**：充分利用默认值、历史数据和动作流，让用户只记录发生的变化，尽量减少输入和思考。

最终，这个工具的定位不是一个复杂的数据库，而更像是一个**能够陪伴每只爬宠整个生命周期的数字饲养日志**：每天打开几秒钟即可完成记录，系统持续整理历史、计算趋势并在合适的时间提醒下一步需要做的事情。这样的设计更符合长期自用的目标，也为未来扩展保留了足够的空间。

# 二十一、当前基础代码启动方式

当前仓库已包含最小可运行的前后端骨架：

* `backend`：FastAPI 后端，提供 `/` 和 `/health` 接口。
* `frontend`：React + Vite 前端，会请求后端 `/health` 并展示连接状态。

## 后端

```powershell
.\.venv\Scripts\Activate.ps1
uvicorn backend.main:app --host 127.0.0.1 --port 8000 --reload
```

访问：

```text
http://127.0.0.1:8000
http://127.0.0.1:8000/docs
```

## 前端

```powershell
cd frontend
npm run dev
```

访问：

```text
http://127.0.0.1:5173
```
